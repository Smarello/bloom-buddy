## User Story

**Epic:** EP-004 — Sicurezza e Accessibilità
**Priorità:** MEDIUM | **Story Points:** 2

**Story**
As operatore del sistema,
I want to limit the number of analysis requests per IP address,
so that the service is protected from abuse and Gemini API costs stay under control.

**Criteri di Accettazione**
- [ ] Rate limiting attivo sulle Route Handlers di analisi (max 10 richieste/minuto per IP)
- [ ] Risposta HTTP 429 con messaggio user-friendly quando il limite è superato
- [ ] Il contatore si resetta automaticamente dopo il periodo di timeout

---

## Soluzione Tecnica

Implementazione di un rate limiter in-memory con una `Map` singleton (`Map<string, { contatore: number; scadenzaMs: number }>`) nel modulo `src/lib/sicurezza/rate-limiter.ts`. La logica è integrata direttamente nella route handler `/api/analyze` esistente, senza introdurre middleware Next.js globale (per evitare di coinvolgere route non interessate). L'IP viene estratto dagli header della request tramite `x-forwarded-for` (Vercel) con fallback a `127.0.0.1` in locale. Lo store in-memory è sufficiente per l'MVP su istanza singola Vercel; in futuro la firma dell'interfaccia permetterà di migrare a Redis/KV senza modificare la route.

- Il rate limiter espone `verificaRateLimit(ip: string): { consentito: boolean; secondiRimanenti: number }` e `resetRateLimiter()` (solo per test)
- La finestra temporale è di 60 secondi con massimo 10 richieste; i contatori scaduti vengono rimossi lazy all'accesso
- La route handler estrae l'IP dagli header, chiama `verificaRateLimit` prima di qualsiasi altra elaborazione e risponde con HTTP 429 e messaggio user-friendly se il limite è superato

---

## Strategia di Test

Test unitari puri sul modulo rate limiter per coprire la logica del contatore, della scadenza e del reset; test di integrazione sulla route handler per verificare il comportamento HTTP 429 e la trasparenza per le richieste nei limiti; uso di `resetRateLimiter()` in `beforeEach` per garantire isolamento tra test.

- **Unit test** `rate-limiter.test.ts`: consentito sotto limite, bloccato al superamento, reset automatico per scadenza, `resetRateLimiter()` resetta lo stato
- **Integration test** `analyze.test.ts` (aggiunta casi): 429 all'undicesima richiesta dallo stesso IP, 200 per IP diversi indipendenti, messaggio user-friendly nel corpo della risposta 429
- **Unit test** estrazione IP: header `x-forwarded-for` valorizzato, assente (fallback), multipli IP nella catena (primo valore)
