## User Story

**Epic:** EP-003 — Analisi Pianta e Visualizzazione Risultati
**Priorità:** HIGH | **Story Points:** 5

**Story**
As Giulia,
I want to upload a photo of my suffering Pothos and receive the plant identification, health diagnosis and personalized care tips,
so that I understand what species it is, what's wrong and what to do to save it.

**Criteri di Accettazione**
- [ ] Route Handler POST `/api/analyze` riceve l'immagine compressa e invoca Google Gemini API con prompt strutturato
- [ ] La risposta include: nome comune, nome scientifico, livello di confidenza, valutazione salute, consigli di cura personalizzati, info generali (annaffiatura, luce, temperatura, umidità)
- [ ] Scheda risultato mostra immagine caricata, nome pianta, indicatore visivo stato di salute (scala colore verde/giallo/rosso) con descrizione testuale
- [ ] Consigli di cura mostrati in linguaggio semplice e incoraggiante
- [ ] Informazioni generali di cura della specie visibili nella scheda
- [ ] API key Gemini conservata esclusivamente server-side in variabili d'ambiente
- [ ] Pulsante "Nuova analisi" per ricominciare con un'altra foto

## Soluzione Tecnica

L'implementazione segue l'architettura full-stack di Next.js 15 App Router: un layer server-side (Route Handler + servizio AI) riceve l'immagine, chiama Gemini, parsifica la risposta nel tipo `PlantAnalysis` già definito, e la ritorna al client; il frontend gestisce la navigazione verso la pagina `/analysis` e renderizza la scheda risultato. Il tipo `PlantAnalysis` in `src/types/analysis.ts` è riutilizzato senza modifiche.

- **Layer AI server-side** (`src/lib/ai/`): client Gemini configurato da env, prompt che richiede output JSON strutturato con schema corrispondente a `PlantAnalysis`, funzione di parsing con gestione fallback per risposte malformate o confidenza bassa
- **Route Handler** (`src/app/api/analyze/route.ts`): riceve `multipart/form-data`, valida tipo e dimensione file server-side, converte in base64 per Gemini, ritorna `PlantAnalysis` serializzato o errore tipizzato
- **Frontend flow**: il bottone "Analizza ora" in `UploadZone` invia `fileCompresso` alla Route Handler tramite un hook `useAnalysis`, poi naviga verso `/analysis` passando il risultato via `sessionStorage` (approccio scelto per evitare URL troppo lunghi con base64)
- **Pagina risultati** (`src/app/analysis/page.tsx`): componente client che legge `PlantAnalysis` da `sessionStorage` e compone la scheda con `AnalysisResult`, `HealthIndicator`, `CareTipsList`, `CareInfoGrid`

## Strategia di Test

Test unitari sul layer AI (parsing) e test di integrazione sulla Route Handler; test di componente per la scheda risultato con tutti e quattro gli stati di salute; focus sulla robustezza del parsing Gemini che è il punto di fallimento più critico.

- Test unitari per `parse-response.ts`: risposta JSON valida, risposta testuale non parsabile, confidenza bassa, immagine non riconosciuta come pianta
- Test di integrazione per `POST /api/analyze`: upload file valido, file non immagine (HTTP 400), API key mancante (HTTP 500), risposta Gemini malformata (errore graceful)
- Test componente `AnalysisResult`: render per tutti e quattro gli stati salute (`excellent`, `good`, `fair`, `poor`), assenza dati (redirect a homepage)
