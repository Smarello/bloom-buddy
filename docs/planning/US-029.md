## Soluzione Tecnica

Si introduce il modello `Collezione` come entitĂ  first-class nel database, eliminando il raggruppamento in-memory per nome pianta. Il campo `collezioneId` su `Analisi` Ă¨ nullable per gestire eventuali analisi orfane. La migrazione dati avviene in SQL dentro il file di migrazione Prisma per garantire atomicitĂ .

- **Schema**: nuovo modello `Collezione` (id, nome, nomeScientifico, utenteId, createdAt) + FK `collezioneId` su `Analisi`
- **Migrazione SQL**: crea collezioni raggruppando per `(utenteId, LOWER(nomeComune))`, prende `nomeScientifico` dall'analisi piĂą recente per gruppo, poi aggiorna `collezioneId` su tutte le analisi
- **Backend**: POST `/api/collezione` crea Collezione + Analisi in transazione Prisma
- **Frontend**: pagina lista usa `prisma.collezione.findMany` con `_count` e `include`; route dettaglio rinominata da `[nomePianta]` a `[idCollezione]` con query per ID
- **Eliminazioni**: `recuperaAnalisiPerPianta.ts` (la type guard `isPlantAnalysis` va estratta in utility condiviso), route API `pianta/[nomePianta]`, raggruppamento in-memory `PiantaRaggruppata`
