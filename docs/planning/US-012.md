## User Story

**Epic:** EP-005 — Autenticazione e Account Utente
**Priorità:** HIGH | **Story Points:** 2

**Story**
Come Giulia,
voglio accedere con le mie credenziali,
così da ritrovare la mia collezione di piante da qualsiasi dispositivo.

**Criteri di Accettazione**
- L'utente può effettuare il login con email e password tramite un form dedicato
- Credenziali errate mostrano un messaggio di errore non tecnico e incoraggiante
- La sessione persiste al refresh della pagina (cookie HttpOnly o sessione server-side)
- L'utente può effettuare il logout e viene reindirizzato alla home

---

## Soluzione Tecnica

La soluzione è simmetrica alla registrazione (US-011): si aggiungono un Route Handler di login (`POST /api/auth/accesso`) che valida le credenziali con `bcrypt.compare` e setta la sessione iron-session, e un Route Handler di logout (`POST /api/auth/logout`) che distrugge la sessione e reindirizza alla home. La Navbar diventa un async Server Component che legge la sessione server-side tramite un nuovo helper `ottieniSessioneServer()` (basato su `cookies()` di `next/headers`) e mostra condizionalmente "Accedi" / "Registrati" vs email utente + pulsante "Esci". Il pulsante "Esci" è implementato come form HTML nativo (`action="/api/auth/logout" method="post"`) per funzionare anche senza JavaScript.

- **Verifica credenziali:** `bcrypt.compare(password, utente.passwordHash)` — risposta 401 con messaggio incoraggiante in caso di mancata corrispondenza
- **Ricerca utente:** `prisma.user.findUnique({ where: { email } })` — utente non trovato trattato identicamente a password errata (sicurezza: no user enumeration)
- **Sessione server-side:** `ottieniSessioneServer()` usa `getIronSession(await cookies(), OPZIONI_SESSIONE)` per Server Components; nessuna tabella di sessione aggiuntiva
- **Logout:** `sessione.destroy()` + `NextResponse.redirect('/')` — nessun dato persiste nel cookie dopo il logout

## Strategia di Test

Test unitari Vitest per i Route Handler (mock Prisma e bcryptjs) e test React Testing Library per il form della pagina `/accesso`; il comportamento condizionale della Navbar è verificato con un test leggero che mocka `ottieniSessioneServer`.

- Test accesso: credenziali errate → 401 con messaggio non tecnico; input mancante → 400; login valido → 200 + sessione popolata
- Test logout: sessione distrutta, risposta di redirect corretta
- Test form `/accesso`: rendering, submit valido (mock fetch), visualizzazione errore non tecnico, stato disabilitato button durante submit
